# Feature Specification: Phase 5 Known Limitations Fix

**Feature Branch**: `006-known-limitations-fix`
**Created**: 2026-06-06
**Status**: Draft
**Input**: Fix three accepted limitations from Phase 5: OCR image attachment (US2-3), real token/generation metrics (US3-4), and ToolsTab.css file length.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Formal OCR Image Attachment (Priority: P1)

A plant engineer clicks an OCR-processed image in the gallery to discuss it in chat. The image is formally attached to the first chat message so the backend can access and process the actual file — not just a text reference to its filename.

**Why this priority**: This is a deferred user story (US2-3) that directly impacts the quality of AI responses about OCR images. Without formal attachment, the AI cannot see the image and can only respond about the filename.

**Independent Test**: Upload an image in the OCR tab, process it, click the image to open in chat. Verify the image appears as an attachment (not just filename text) and the AI can reference the actual image content.

**Acceptance Scenarios**:

1. **Given** an OCR-processed image exists in the gallery, **When** the user clicks "Open in Chat", **Then** a new chat session is created with the image formally attached to the first message, and the backend receives the file reference for processing.
2. **Given** the user opens an OCR image in chat, **When** the AI responds, **Then** the AI can reference and describe the actual image content (not just the filename).
3. **Given** an OCR image is opened in chat, **When** the chat session loads, **Then** the image thumbnail is visible as an attachment on the user message (consistent with how manual file attachments appear in chat).

---

### User Story 2 - Real Token Count and Generation Time (Priority: P2)

A plant engineer reviews an analysis result and sees accurate token usage and actual generation time — not estimates — allowing them to understand real resource consumption and model performance.

**Why this priority**: Estimated metrics are misleading. Real metrics enable accurate performance monitoring and trust in the displayed information.

**Independent Test**: Run an analysis query, verify the displayed token count matches the actual number of generated tokens (not characters/4), and verify generation time reflects actual wall-clock time from the backend.

**Acceptance Scenarios**:

1. **Given** the user has completed an analysis query, **When** viewing the analysis results, **Then** the token count displays the actual number of tokens generated by the model (prompt tokens + completion tokens), not a character-based estimate.
2. **Given** the user has completed an analysis query, **When** viewing the analysis results, **Then** the generation time displays the actual time taken by the model to generate the response, measured from the backend.
3. **Given** the backend does not return usage metadata for a response, **When** the analysis results are displayed, **Then** the system falls back to the current estimation method with a visual indicator that the values are estimated.

---

### User Story 3 - ToolsTab.css File Length Reduction (Priority: P3)

The ToolsTab.css file exceeds the project's 400-line limit. Styles are extracted into component-specific files to maintain the one-concept-per-file principle.

**Why this priority**: Code quality and maintainability. No user-facing behavior change — purely a refactor.

**Independent Test**: Verify ToolsTab.css is under 400 lines, extracted styles render identically, and no visual regression occurs.

**Acceptance Scenarios**:

1. **Given** the ToolsTab.css file is refactored, **When** the Tools tab renders, **Then** all visual elements (gauge cards, audio transcriber, RAGAS evaluator, history table) appear identically to before the refactor.

### Edge Cases

- What happens when an OCR image file has been deleted from storage but still appears in the gallery? The chat should handle the missing file gracefully with a clear error.
- What happens when the backend returns partial usage metadata (e.g., token count but no generation time)? The frontend should display available real values and estimate missing ones.
- What happens when an SSE connection drops before the `done` event with usage metadata? The frontend should fall back to estimated values.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The OCR "Open in Chat" action MUST attach the image file to the chat message so the backend can access the actual file content, not just the filename.
- **FR-002**: The backend chat endpoint MUST accept and process file references passed from the OCR tab's "Open in Chat" action.
- **FR-003**: The image attachment MUST be visible as a thumbnail on the user message in the chat interface, consistent with manual file attachment behavior.
- **FR-004**: The backend MUST include actual token usage metrics (prompt tokens, completion tokens, total tokens) in the streaming response when available from the LLM provider.
- **FR-005**: The backend MUST include actual generation time (wall-clock time from request to last token) in the streaming response.
- **FR-006**: The analysis display MUST show real token counts and generation time when provided by the backend, and fall back to estimated values with a visual indicator when real data is unavailable.
- **FR-007**: ToolsTab.css MUST be reduced to under 400 lines by extracting cohesive style blocks into component-specific CSS files.
- **FR-008**: Extracted CSS files MUST follow the one-concept-per-file principle and be imported by their respective components.
- **FR-009**: All visual elements in the Tools tab MUST render identically after CSS extraction (no visual regression).

### Key Entities

- **Chat Message Attachment**: Represents a file attached to a chat message. Has a file reference (backend file ID), display name, and thumbnail. Already exists in the system for manual uploads — this work extends it to OCR-initiated flows.
- **Usage Metadata**: Token usage and timing data for an AI response. Includes prompt token count, completion token count, total token count, and generation duration in milliseconds. Returned alongside the completed AI response.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: OCR images opened in chat are processed by the AI with reference to actual file content, verifiable by the AI describing visible image features (not just the filename).
- **SC-002**: Token counts displayed in the Analysis tab match actual LLM output token counts within a 5% margin, replacing the previous character-division estimate.
- **SC-003**: Generation time displayed in the Analysis tab reflects actual backend-measured duration, not a sum of thinking step durations.
- **SC-004**: ToolsTab.css is under 400 lines with all extracted styles in separate, component-specific CSS files.
- **SC-005**: No visual regression in the Tools tab after CSS extraction — all components render identically to before the change.

## Assumptions

- The backend chat endpoint already supports receiving file IDs via the `attached_files` parameter — only frontend wiring is needed for LIM-1.
- The LLM provider (Ollama) returns usage metadata in its response that can be captured and propagated — if not available for a specific request, estimated values serve as fallback.
- The OCR gallery images are stored as file records in the backend with accessible file IDs — the existing `/api/files/{file_id}/content` endpoint can serve the image binary.
- Audio transcriber styles are the best extraction candidate for LIM-3 since `AudioTranscriber.tsx` is already a separate component with ~166 lines of dedicated styles.
- CSS extraction for LIM-3 will not change any class names — only move existing rules to new files.
